# Distribution and Release Strategy

This document defines how the `meta-workspace` project should be installed and released.

The release unit is the Rust `mw` binary. The deployable workspace template is embedded into that binary, so a given `mw` version owns the template version it materializes with `mw init`.

## Goals

- Keep the base workspace runtime-free.
- Keep distribution npm-free.
- Make the first usable path simple for authorized users.
- Add official release automation only when the project is ready for stable releases.
- Keep binary and template versions tied together.

## Current path: private source install

For now, install `mw` from an authorized Git checkout and Cargo. This avoids release machinery while the project is still stabilizing.

Prerequisites:

- Rust toolchain with Cargo.
- Access to the repository.

Install:

```bash
mkdir -p "$HOME/src"
git clone git@github.com:BarraDev/meta-workspace.git "$HOME/src/meta-workspace"
cargo install --path "$HOME/src/meta-workspace/tooling" --locked --force
mw --version
```

Update an existing install:

```bash
git -C "$HOME/src/meta-workspace" pull --ff-only
cargo install --path "$HOME/src/meta-workspace/tooling" --locked --force
mw --version
```

Create a workspace after installing `mw`:

```bash
mkdir -p "$HOME/work-root/example-meta-workspace"
cd "$HOME/work-root/example-meta-workspace"
mw init --company-name "Example Company" --company-id example-company
mw doctor
```

This uses `cargo install --path` instead of `cargo install --git` because the crate currently lives in `tooling/`, and the committed lockfile also lives there. The path install keeps the current repository shape and honors `tooling/Cargo.lock` with `--locked`.

## Internal tags before official releases

If a stable internal checkpoint is needed before official release automation exists:

1. Run the gates:

   ```bash
   cd tooling
   cargo fmt --check
   cargo clippy --all-targets -- -D warnings
   cargo test
   ```

2. Tag the repository with a clearly internal version, for example:

   ```bash
   git tag v0.1.0-internal.1
   git push origin v0.1.0-internal.1
   ```

3. Install from a checkout at that tag:

   ```bash
   git -C "$HOME/src/meta-workspace" fetch --tags
   git -C "$HOME/src/meta-workspace" checkout v0.1.0-internal.1
   cargo install --path "$HOME/src/meta-workspace/tooling" --locked --force
   ```

Return the checkout to `main` when continuing development:

```bash
git -C "$HOME/src/meta-workspace" checkout main
git -C "$HOME/src/meta-workspace" pull --ff-only
```

## Official release path: cargo-dist later

When the project is ready for official releases, add `cargo-dist` and use GitHub Releases as the primary binary distribution channel.

Expected official channels:

- GitHub Releases with prebuilt archives and checksums.
- `cargo-binstall` metadata so users can fetch prebuilt binaries.
- `cargo install meta-workspace` from crates.io, if publishing is approved.
- Shell installer generated by `cargo-dist`.
- Homebrew tap later, after release cadence is stable.

Readiness checklist before adding `cargo-dist`:

- Confirm repository and release location.
- Decide whether and when to publish `meta-workspace` to crates.io.
- Confirm crate ownership, license, README, categories, keywords, and repository metadata.
- Confirm versioning policy for the embedded template and `workspace.yaml` schema.
- Decide supported target triples for Linux, macOS, and Windows.
- Add release CI triggered by tags.
- Verify generated archives, checksums, and installer output on a dry run.

## Channel progression

| Stage | Channel | Use when |
|---|---|---|
| 4A | private checkout + `cargo install --path tooling --locked` | current development and early authorized use |
| 4B | internal Git tags + path install | stable internal checkpoints before official automation |
| 4C | GitHub Releases via `cargo-dist` | official binary releases |
| 4D | crates.io / `cargo-binstall` / Homebrew | broader distribution after release cadence is proven |

## Non-goals for now

- No npm package for the base distribution.
- No network fetch of the template during `mw init`; the template stays embedded in the binary.
- No `mw eject` implementation until there is real demand.
